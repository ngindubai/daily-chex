Design Direction

Military ops control room meets DeWalt

  • Palette: Jet black background (#0a0a0a), carbon dark surfaces (#111113), DeWalt yellow (#facc15) accents, signal green (#22c55e) for pass, alert red (#ef4444) for fail/critical
  • Typography: Bold industrial sans-serif — Rubik or similar with tight tracking for headers, clean readable body text
  • Feel: Dark CRT glow, scanline textures, dashboard HUD elements, data-dense cards, subtle yellow outlines on active elements
  • Animations: Smooth spring-based transitions, elements that slide/scale in with weight, haptic-feeling button presses (scale-down + shadow shift), staggered card reveals, pulse effects on live data
  • Map: Dark-themed map tiles, yellow/green/red markers, pulsing "live" dots for active users, connection lines between operator and asset
  • Cards: Overdue checks = red glow border, all-clear = green tick + subtle green glow, pending = amber pulse
  • Mobile: Touch-optimised, large tap targets, swipe actions, bottom-tab navigation

How GPS tracking works in Daily-Chex

  1. On QR scan / check start: App calls navigator.geolocation.getCurrentPosition() with enableHighAccuracy: true. Captures lat, lng, accuracy to the check record.
  2. On check complete: Second GPS fix recorded. Now we have start + end coordinates.
  3. Asset location vs operator location: The map shows where the operator was when they scanned the QR code. If the operator is within the site boundary, it confirms "on-site check". If not, the dashboard flags "off-site check" with an amber indicator.
  4. Live map view: Dashboard shows a Leaflet map with markers for recent checks (last 24h). Yellow pin = check done today, green = all-pass, red = defect found. Clicking a marker shows the check details.
  5. No background tracking: We only capture GPS at the moment of scan/check action. No continuous tracking. This is privacy-friendly and avoids battery drain.

Build Steps — 10 Phases

Each step is self-contained, testable, and deployable. Tell me "begin step X" to start any phase.

1
Project Scaffold & Design System
Foundation Frontend

Set up the project structure, install dependencies, and build the design system — the visual foundation everything sits on.

What gets built

  • React + Vite + TypeScript project in projects/chex/app/
  • PWA manifest + service worker setup
  • Tailwind CSS with the Daily-Chex DeWalt theme (black/yellow tokens)
  • Design system: buttons, cards, badges, inputs, status indicators
  • Layout shell: sidebar nav (desktop), bottom tab bar (mobile)
  • Page transitions and micro-interactions (spring animations, button press effects)
  • Dark military-control aesthetic applied globally
  • Responsive breakpoints: mobile-first, tablet, desktop

Deliverable

A running app shell with no functionality but the full visual language — someone can open it and say "that looks like Daily-Chex".

Dependencies: None — this is the starting point.
2
Backend & Database Foundation
Foundation Backend

The API server, database schema, and core data model.

What gets built

  • Express (or Hono) API server in projects/chex/server/
  • PostgreSQL schema — all tables from the implementation plan: companies, sites, teams, people, assets, checks, check_items, check_photos, defects, check_templates, check_template_items
  • Drizzle ORM setup with migrations
  • Seed script with realistic data (SAW Utilities example from the plan)
  • Health check endpoint
  • CORS and basic middleware (JSON parsing, error handling)

Deliverable

A running API server at localhost:3200 with a seeded database. Can query /api/health and see data.

Dependencies: None — can be built in parallel with Step 1.
3
Auth System (PIN + Email)
Backend Frontend

PIN-based login for operators, email/password for managers, JWT tokens, device trust.

What gets built

  • API: POST /api/auth/pin — PIN login, returns JWT
  • API: POST /api/auth/login — email/password login for managers
  • API: GET /api/auth/me — current user + company + team + site context
  • JWT middleware with role-based guards (operator / supervisor / manager / admin)
  • PIN hashing (bcrypt) and uniqueness within company
  • Frontend: PIN entry screen (big number pad, industrial feel)
  • Frontend: Email login screen (for managers/supervisors)
  • Frontend: Auth context provider, protected routes
  • Device trust: store company token in localStorage for quick re-auth

Deliverable

Operators can enter a 4-digit PIN and land on their home screen. Managers can log in with email and see the dashboard shell.

Depends on: Step 1 (frontend), Step 2 (backend).
4
Site, Team & People Management
Backend Frontend

The organisational hierarchy — sites, teams within sites, people assigned to teams. This is the backbone that everything else hangs off.

What gets built

  • API: Full CRUD for sites, teams, people + assignment endpoints
  • Frontend — Sites page: list of active sites, site cards showing team count + asset count + recent activity
  • Frontend — Site detail: teams at this site, people in each team, quick-add team/person modals
  • Frontend — People directory: all operators/supervisors, their current team/site assignment, search + filter
  • Frontend — Team view: members list, team leader badge, assign/unassign people
  • Drag-to-reassign: move a person between teams within a site
  • Cross-site transfer for people
  • Archive sites (preserve history, hide from active views)

Deliverable

A manager can create sites, add teams to sites, add people to teams, and see the organisational structure in the dashboard.

Depends on: Step 2 (database), Step 3 (auth for access control).
5
Asset Register & QR Codes
Backend Frontend QR

The asset register (vehicles, trailers, plant), QR code generation, QR scanning, and asset transfer between sites.

What gets built

  • API: Asset CRUD, QR lookup (GET /api/assets/qr/:code), asset transfer
  • QR code generation: unique CHX-XXXXXXXX codes
  • QR label PDF: printable sticker sheets (62mm × 30mm labels)
  • Frontend — Asset register: filterable list (by site, type, status, supplier), search by name/reg/plant ID
  • Frontend — Asset detail: full info card, current site, supplier, calibration dates, photo, check history placeholder
  • Frontend — Add asset form: type selector (vehicle/trailer/plant), conditional fields (reg for vehicles, weight class, supplier, etc.)
  • Frontend — QR scanner: camera-based scanning using html5-qrcode, resolves to asset detail
  • Frontend — QR sticker preview: shows the label before printing, batch print for multiple assets
  • Asset transfer: move to different site with timestamp logging

Deliverable

Manager can add assets, generate QR stickers, print them. Operator can scan a QR code with their phone and see the asset.

Depends on: Step 4 (sites for asset assignment).
6
Check Templates & Check Flow
Backend Frontend Core Feature

The heart of the app. The three check templates, the guided check flow, pass/fail/NA for each item, defect reporting, digital signatures, photo capture, and GPS location capture.

What gets built

  • API: Check lifecycle — create, add items, complete, sign, countersign
  • API: Check templates with items, sections, conditional display rules
  • Three templates seeded: Vehicle & Trailer Weekly, PUWER Weekly, Vehicle Daily
  • Frontend — Check flow (guided):
    1. Scan QR / select asset → confirm identity + site + team
    2. GPS capture at start (Browser Geolocation API)
    3. Mileage entry (if vehicle)
    4. Checklist: pass/fail/NA per item, with expandable notes + camera on fail
    5. Conditional sections (7.5T+, towing/lifting) based on asset properties
    6. Defect summary screen with severity + description + photos
    7. Digital signature pad (signature_pad library)
    8. GPS capture at completion
    9. Submit with confirmation screen
  • Frontend — PUWER weekly grid: the different flow for plant inspections (site-level, Mon–Sun grid per asset)
  • Photo capture: compress to 1200px/200KB, store locally until sync
  • Auto-save in progress to prevent data loss

Deliverable

An operator can scan a QR code, run through the full check, take photos of defects, sign off, and submit. The PUWER weekly grid matches the paper form experience.

Depends on: Step 5 (assets + QR scanning).
7
Defect Management
Backend Frontend

Defect lifecycle — from report through to resolution. Cards showing severity, assignment, status.

What gets built

  • API: Defect CRUD, status transitions (open → in_progress → resolved / accepted)
  • Auto-create defects from failed check items
  • Critical defect auto-flags asset as defective
  • Frontend — Defect queue: card-based view sorted by severity, filterable by site/team/asset
  • Frontend — Defect detail: description, photos, linked check, action taken, resolution log
  • Frontend — Defect resolution: supervisor marks as resolved, adds action taken notes
  • Overdue defect highlighting (open > 24h = amber, > 72h = red glow)

Deliverable

Defects flow from checks into a management queue. Supervisors can triage, resolve, and track defect lifecycle.

Depends on: Step 6 (checks create defects).
8
Dashboard & Map View
Frontend Map Key Feature

The military-control dashboard. Interactive map showing where checks happened, status cards, overdue alerts, site overview.

What gets built

  • API: Dashboard stats endpoints (overview, per-site, overdue, calibrations)
  • Frontend — Command Overview (manager):
    • Top stats row: total assets, checks today, open defects, overdue count, calibration warnings
    • Interactive map (Leaflet + dark tiles): markers for every check in the last 24h, colour-coded (green=pass, red=defect, yellow=in-progress)
    • Map cluster view at wide zoom, individual markers at close zoom
    • Click marker → popup with check details, operator name, timestamp, pass/fail result
    • Site boundary polygons drawn on the map (optional GPS fencing)
    • Pulsing dot for "live" — checks submitted in the last 5 minutes
  • Frontend — Status cards grid:
    • Cards per site: name, check completion rate (bar), defect count, team count
    • Overdue checks card: red-bordered, lists assets + how long overdue
    • Calibration card: amber, upcoming calibrations with countdown
    • Defect summary card: severity breakdown (critical/high/medium/low)
  • Frontend — Supervisor site view: same concepts but scoped to their site, with team filter
  • Frontend — Operator home: big "Scan QR" button, their recent checks, assigned assets needing checks today

Deliverable

A manager opens the dashboard and gets a real-time ops-centre view: map with pins showing every check, cards showing what's overdue, what has defects, what's coming up for calibration.

Depends on: Step 6 (checks with GPS data), Step 7 (defect status for cards).
9
Offline / PWA & Sync
Frontend Backend

Make the whole thing work without internet. Service worker, IndexedDB caching, offline QR scanning, sync queue.

What gets built

  • Service worker: caches app shell, templates, asset register
  • IndexedDB (via Dexie.js): local asset register, offline check storage, photo blob storage
  • Offline QR lookup: matches against cached asset register
  • Sync queue: checks completed offline queue for upload when connectivity returns
  • Conflict resolution: last-write-wins for checks, server-wins for defect resolution
  • Sync status indicator: "2 checks pending sync" badge in header
  • Photo compression pipeline: resize to 1200px max, ~200KB, store in IndexedDB
  • PWA install prompt: "Add Daily-Chex to Home Screen"
  • Background sync using Service Worker Background Sync API where available

Deliverable

An operator in a trench with no signal can scan a QR code, complete a full check with photos, sign off, and it syncs when they come back to the surface.

Depends on: Step 6 (check flow to take offline), Step 5 (asset register to cache).
10
Polish, Testing & Launch Prep
QA Launch

Cross-browser testing, edge case handling, performance pass, and getting ready for real-world use.

What gets built

  • Cross-browser testing: Chrome, Safari mobile, Samsung Internet, Firefox
  • Real QR sticker testing with printed labels on actual equipment
  • Performance audit: Lighthouse scores, load time on 3G, offline boot time
  • Edge cases from the implementation plan (phone dying mid-check, duplicate scans, etc.)
  • Error states and empty states for every screen
  • Loading skeletons (not spinners — skeletons feel faster)
  • Haptic feedback on mobile (navigator.vibrate for button presses)
  • Final animation polish pass: staggered reveals, page transitions, micro-interactions
  • Logo integration (once client uploads)
  • Deployment to production hosting
  • UAT with client operators on a real site

Deliverable

A production-ready app that works on every phone, in every signal condition, and feels slick enough that tradesmen actually want to use it.

Depends on: All previous steps.
